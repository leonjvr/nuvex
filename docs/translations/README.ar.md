## التثبيت

### المتطلبات الأساسية

| الأداة | مطلوبة | ملاحظات |
|--------|---------|---------|
| **Node.js** | >= 22.0.0 | وحدات ES، `fetch()`، `crypto.subtle`. [تنزيل](https://nodejs.org) |
| **سلسلة أدوات C/C++** | لبناء المصدر فقط | `better-sqlite3` و`argon2` يُجمِّعان إضافات أصلية |
| **Docker** | >= 24 (اختياري) | لنشر Docker فقط |

تثبيت Node.js 22: Ubuntu/Debian (`curl -fsSL https://deb.nodesource.com/setup_22.x | sudo -E bash - && sudo apt-get install -y nodejs`)، macOS (`brew install node@22`)، Windows (`winget install OpenJS.NodeJS.LTS`).

تثبيت أدوات C/C++: Ubuntu (`sudo apt-get install -y python3 make g++ build-essential`)، macOS (`xcode-select --install`)، Windows (`npm install --global windows-build-tools`).

### الخيار أ — Docker (موصى به)

```bash
git clone https://github.com/GoetzKohlberg/sidjua.git
cd sidjua
docker compose up -d
```

```bash
# عرض مفتاح API المُنشأ تلقائيًا
docker compose exec sidjua cat /app/.system/api-key

# تهيئة الحوكمة
docker compose exec sidjua sidjua apply --verbose

# فحص صحة النظام
docker compose exec sidjua sidjua selftest
```

يدعم **linux/amd64** و**linux/arm64** (Raspberry Pi، Apple Silicon).

### الخيار ب — التثبيت العام عبر npm

```bash
npm install -g sidjua
sidjua init          # إعداد تفاعلي من 3 خطوات
sidjua chat guide    # مرشد ذكاء اصطناعي بدون إعداد (لا يحتاج مفتاح API)
```

### الخيار ج — بناء من المصدر

```bash
git clone https://github.com/GoetzKohlberg/sidjua.git
cd sidjua
npm ci && npm run build
node dist/index.js --help
```

## لماذا SIDJUA؟

تعتمد كل أطر عمل وكلاء الذكاء الاصطناعي اليوم على الافتراض المعطوب ذاته: أنك
تستطيع الوثوق بالذكاء الاصطناعي ليتبع قواعده الخاصة.

**المشكلة في الحوكمة القائمة على التوجيهات:**

تمنح الوكيل توجيهًا للنظام يقول "لا تصل أبدًا إلى بيانات PII الخاصة بالعملاء." يقرأ
الوكيل التعليمات. كما يقرأ رسالة المستخدم الذي يطلب منه سحب سجل مدفوعات جون سميث.
يقرر الوكيل — بنفسه — ما إذا كان سيمتثل. هذه ليست حوكمة. هذه توصية صاغها شخص قوي الإرادة.

**SIDJUA مختلف.**

تجلس الحوكمة **خارج** الوكيل. كل إجراء يمر عبر خط أنابيب تطبيق مؤلف من 5 مراحل
**قبل** التنفيذ. تُعرِّف القواعد في YAML. يُطبِّقها النظام. لا يحصل الوكيل على فرصة
لتحديد ما إذا كان سيتبعها، لأن الفحص يحدث قبل أن يتصرف الوكيل.

هذه حوكمة بالبنية المعمارية — لا بالتوجيهات، ولا بالضبط الدقيق، ولا بالأمل.

## كيف يعمل

يُغلِّف SIDJUA وكلاءك في طبقة حوكمة خارجية. لا يحدث استدعاء LLM للوكيل حتى
يجتاز الإجراء المقترح خط أنابيب تطبيق من 5 مراحل:

**المرحلة 1 — المحظور:** تُرفَض الإجراءات المحظورة فورًا. لا استدعاء LLM، ولا إدخال سجل
مُوسَّم بـ"مسموح"، ولا فرصة ثانية. إذا كان الإجراء في قائمة المحظورات، فإنه يتوقف هنا.

**المرحلة 2 — الموافقة:** تُحتجز الإجراءات التي تتطلب موافقة بشرية للمراجعة قبل
التنفيذ. ينتظر الوكيل. يقرر الإنسان.

**المرحلة 3 — الميزانية:** كل مهمة تعمل مقابل حدود التكلفة الفورية. تُطبَّق ميزانيات
لكل مهمة ولكل وكيل. عند الوصول إلى الحد، تُلغى المهمة — لا تُعلَّم فقط، لا تُسجَّل للمراجعة،
بل *تُلغى*.

**المرحلة 4 — التصنيف:** تُفحَص البيانات التي تعبر حدود الأقسام مقابل قواعد التصنيف.
وكيل من المستوى 2 لا يستطيع الوصول إلى بيانات SECRET. الوكيل في القسم أ لا يستطيع
قراءة أسرار القسم ب.

**المرحلة 5 — السياسة:** قواعد تنظيمية مخصصة، مُطبَّقة هيكليًا. حدود تكرار استدعاءات
API، وحدود رموز المخرجات، وقيود النافذة الزمنية.

يعمل خط الأنابيب بأكمله قبل تنفيذ أي إجراء. لا يوجد وضع "تسجيل ومراجعة لاحقًا"
للعمليات الحرجة من الناحية الحوكمية.

### ملف تكوين واحد

تعيش منظمة وكلاءك بأكملها في ملف `divisions.yaml` واحد:

```yaml
divisions:
  - name: engineering
    agents:
      - name: research-agent
        provider: anthropic
        model: claude-haiku-4-5-20251001
        tier: 2
        budget:
          per_task_usd: 0.50
          per_month_usd: 50.00
    governance:
      rules:
        - no_external_api_calls: true
        - max_tokens_per_response: 4096
        - require_human_approval: [delete, send_email]
```

يقرأ `sidjua apply` هذا الملف ويُوفِّر البنية التحتية الكاملة للوكلاء:
الوكلاء، والأقسام، وRBAC، والتوجيه، وجداول التدقيق، ومسارات الأسرار، وقواعد الحوكمة
— في 10 خطوات قابلة للتكرار.

### بنية الوكيل المعمارية

تُنظَّم الوكلاء في **أقسام** (مجموعات وظيفية) و**مستويات** (مستويات الثقة). وكلاء
المستوى 1 لديهم استقلالية كاملة ضمن مظروف حوكمتهم. وكلاء المستوى 2 يحتاجون إلى
موافقة للعمليات الحساسة. وكلاء المستوى 3 خاضعون للإشراف الكامل. يُطبَّق نظام المستويات
هيكليًا — لا يستطيع الوكيل ترقية نفسه.

```
┌─────────────────────────────────────────────────┐
│                 SIDJUA Platform                 │
│                                                 │
```

## قيود البنية المعمارية

يُطبِّق SIDJUA هذه القيود على مستوى البنية المعمارية — لا يمكن تعطيلها أو تجاوزها
أو تجاهلها من قِبَل الوكلاء:

1. **الحوكمة خارجية**: تُغلِّف طبقة الحوكمة الوكيل. الوكيل ليس لديه وصول إلى كود
   الحوكمة، ولا يمكنه تعديل القواعد، ولا يمكنه اكتشاف ما إذا كانت الحوكمة موجودة.

2. **قبل الإجراء، لا بعده**: يُفحَص كل إجراء قبل التنفيذ. لا يوجد وضع "تسجيل
   ومراجعة لاحقًا" للعمليات الحرجة من الناحية الحوكمية.

3. **التطبيق الهيكلي**: تُطبَّق القواعد بمسارات الكود، لا بالتوجيهات أو تعليمات
   النموذج. لا يستطيع الوكيل "كسر القيود" والخروج من الحوكمة لأن الحوكمة لا تُطبَّق
   كتعليمات للنموذج.

4. **ثبات التدقيق**: سجل الكتابة المسبقة (WAL) يقبل الإضافة فقط مع التحقق من السلامة.
   تُكشَف الإدخالات المُعدَّلة وتُستثنى.

5. **عزل الأقسام**: الوكلاء في الأقسام المختلفة لا يستطيعون الوصول إلى بيانات
   بعضهم، أو أسرارهم، أو قنوات اتصالهم.

## الميزات

### الحوكمة والامتثال

**خط أنابيب ما قبل الإجراء (المرحلة 0)** يعمل قبل كل إجراء للوكيل: فحص المحظورات
→ الموافقة البشرية → تطبيق الميزانية → تصنيف البيانات → السياسة المخصصة. جميع
المراحل الخمس هيكلية — تُنفَّذ في الكود، لا في توجيه الوكيل.

**قواعد البداية الإلزامية** تُشحَن مع كل تثبيت: 10 قواعد حوكمة (`SYS-SEC-001`
حتى `SYS-GOV-002`) لا يمكن إزالتها أو إضعافها من خلال تكوين المستخدم. تُوسِّع
القواعد المخصصة البداية؛ لا يمكنها تجاوزها.

**الامتثال لـ EU AI Act** — سجل التدقيق، وإطار التصنيف، وسير عمل الموافقة تتطابق
مباشرة مع متطلبات المادة 9 و12 و17. الموعد النهائي للامتثال في أغسطس 2026 مدمج
في خارطة طريق المنتج.

**تقارير الامتثال** عبر `sidjua audit report/violations/agents/export`:
درجة الامتثال، ودرجات الثقة لكل وكيل، وسجل المخالفات، وتصدير CSV/JSON
للمدققين الخارجيين أو تكامل SIEM.

**سجل الكتابة المسبقة (WAL)** مع التحقق من السلامة: كل قرار حوكمي يُكتَب في سجل
يقبل الإضافة فقط قبل التنفيذ. تُكشَف الإدخالات المُعدَّلة عند القراءة. `sidjua memory recover`
يُعيد التحقق والإصلاح.

### التواصل

الوكلاء لا يستجيبون فقط لاستدعاءات API — بل يشاركون في قنوات اتصال حقيقية.

**البريد الإلكتروني ثنائي الاتجاه** (`sidjua email status/test/threads`): يستقبل
الوكلاء البريد الإلكتروني عبر استطلاع IMAP ويردون عبر SMTP. يحافظ تعيين الخيوط
عبر رؤوس In-Reply-To على تماسك المحادثات. القائمة البيضاء للمُرسِلين، وحدود حجم
الجسم، وإزالة HTML تحمي خط أنابيب الوكيل من المدخلات الضارة.

**بوت بوابة Discord**: واجهة أوامر شاملة عبر `sidjua module install discord`.
يستجيب الوكلاء لرسائل Discord، ويحافظون على خيوط المحادثة، ويُرسِلون إشعارات استباقية.

**تكامل Telegram**: تنبيهات وإشعارات الوكيل عبر بوت Telegram. يدعم نمط المحول
متعدد القنوات Telegram وDiscord وntfy والبريد الإلكتروني بالتوازي.

### العمليات

**أمر Docker واحد** للإنتاج:

```bash
docker run -p 4200:4200 ghcr.io/goetzkohlberg/sidjua:latest
```

يُنشأ مفتاح API تلقائيًا عند أول تشغيل ويُطبَع في سجلات الحاوية.
لا متغيرات بيئة مطلوبة. لا إعداد مطلوب. لا خادم قاعدة بيانات مطلوب —
يستخدم SIDJUA SQLite، ملف قاعدة بيانات واحد لكل وكيل.

**إدارة CLI** — دورة حياة كاملة من ملف ثنائي واحد:

```bash
sidjua init                      # إعداد مساحة العمل التفاعلي (3 خطوات)
sidjua apply                     # التوفير من divisions.yaml
sidjua agent create/list/stop    # دورة حياة الوكيل
sidjua run "task..." --wait      # إرسال مهمة مع تطبيق الحوكمة
sidjua audit report              # تقرير الامتثال
sidjua costs                     # تفصيل التكلفة حسب القسم/الوكيل
sidjua backup create/restore     # إدارة النسخ الاحتياطية المُوقَّعة بـ HMAC
sidjua update                    # تحديث الإصدار مع نسخ احتياطي تلقائي مسبق
sidjua rollback                  # استعادة بنقرة واحدة إلى الإصدار السابق
sidjua email status/test         # إدارة قناة البريد الإلكتروني
sidjua secret set/get/rotate     # إدارة الأسرار المشفرة
sidjua memory import/search      # خط أنابيب المعرفة الدلالية
sidjua selftest                  # فحص صحة النظام (7 فئات، درجة 0-100)
```

**الذاكرة الدلالية** — استيراد المحادثات والمستندات (`sidjua memory import
~/exports/claude-chats.zip`)، والبحث باستخدام التصنيف الهجين للمتجهات + BM25. يدعم
تضمينات Cloudflare Workers AI (مجانية، بدون إعداد) وتضمينات OpenAI الكبيرة
(جودة أعلى لقواعد المعرفة الكبيرة).

**التقطيع التكيفي** — يضبط خط أنابيب الذاكرة تلقائيًا أحجام الأجزاء للبقاء ضمن
حد الرموز لكل نموذج تضمين.

**مرشد بدون إعداد** — `sidjua chat guide` يُشغِّل مساعد ذكاء اصطناعي تفاعلي
بدون أي مفتاح API، مدعوم بـ Cloudflare Workers AI عبر وكيل SIDJUA.
اسأله كيفية إعداد الوكلاء، وتكوين الحوكمة، أو فهم ما حدث في سجل التدقيق.

**النشر المعزول عن الشبكة** — تشغيل منفصل تمامًا عن الإنترنت باستخدام نماذج LLM
المحلية عبر Ollama أو أي نقطة نهاية متوافقة مع OpenAI. لا قياس عن بُعد افتراضيًا.
تقارير الأعطال الاختيارية مع إزالة كاملة لـ PII.

### الأمن

**عزل صندوق الرمل** — تعمل مهارات الوكيل داخل عزل العملية على مستوى نظام التشغيل
عبر bubblewrap (مساحات أسماء مستخدمي Linux). صفر تكلفة إضافية لـ RAM. واجهة
`SandboxProvider` قابلة للتوصيل: `none` للتطوير، `bubblewrap` للإنتاج.

**إدارة الأسرار** — مخزن أسرار مشفر مع RBAC (`sidjua secret
set/get/list/delete/rotate/namespaces`). لا حاجة إلى خزينة خارجية.

**البناء الأمني أولًا** — مجموعة اختبارات داخلية شاملة بالإضافة إلى تحقق مستقل
من قِبَل 2 من المدققين الخارجيين للكود (DeepSeek V3 وxAI Grok). رؤوس أمان،
وحماية CSRF، وتحديد المعدل، وتعقيم المدخلات على كل سطح API. منع حقن SQL مع
الاستعلامات المُعاملية في كل مكان.

**سلامة النسخ الاحتياطية** — أرشيفات نسخ احتياطية مُوقَّعة بـ HMAC مع حماية zip-slip،
ومنع قنابل ZIP، والتحقق من مجموع تدقيق البيان عند الاستعادة.

## الاستيراد من أطر عمل أخرى

```bash
# معاينة ما سيُستورَد — لا تغييرات تُجرى
sidjua import openclaw --dry-run

# استيراد ملفات التكوين + المهارات
sidjua import openclaw --skills
```

تحتفظ وكلاؤك الموجودون بهويتهم، ونماذجهم، ومهاراتهم. يُضيف SIDJUA
الحوكمة، وسجلات التدقيق، وضوابط الميزانية تلقائيًا.

## مرجع التكوين

ملف `divisions.yaml` أدنى مستوى للبدء:

```yaml
organization:
  name: "my-org"
  tier: 1

divisions:
  - name: operations
    tier: 2
    agents:
      - name: ops-agent
        provider: anthropic
        model: claude-haiku-4-5-20251001
        division: operations
        budget:
          per_task_usd: 0.25
          per_month_usd: 25.00

governance:
  stage0:
    enabled: true
    forbidden_actions:
      - delete_database
      - exfiltrate_data
    classification:
      default_level: INTERNAL
      max_agent_level: CONFIDENTIAL
```

يُوفِّر `sidjua apply` البنية التحتية الكاملة من هذا الملف. قم بتشغيله
مرة أخرى بعد التغييرات — إنه مُتماثل الأداء.

راجع [docs/SIDJUA-APPLY-TECH-SPEC-V1.md](docs/SIDJUA-APPLY-TECH-SPEC-V1.md)
للمواصفات الكاملة لجميع خطوات التوفير العشر.

## REST API

تعمل REST API الخاصة بـ SIDJUA على نفس المنفذ مثل لوحة المعلومات:

```bash
sidjua server start --port 4200 --api-key $SIDJUA_API_KEY
```

نقاط النهاية الرئيسية:

```
GET  /api/v1/health          # فحص الصحة العام (بدون مصادقة)
GET  /api/v1/info            # بيانات وصفية للنظام (مع مصادقة)
POST /api/v1/execute/run     # إرسال مهمة
GET  /api/v1/execute/:id/status  # حالة المهمة
GET  /api/v1/execute/:id/result  # نتيجة المهمة
GET  /api/v1/events          # تدفق أحداث SSE
GET  /api/v1/audit/report    # تقرير الامتثال
```

جميع نقاط النهاية باستثناء `/health` تتطلب مصادقة Bearer. إنشاء مفتاح:

```bash
sidjua api-key generate
```

## Docker Compose

```yaml
services:
  sidjua:
    image: ghcr.io/goetzkohlberg/sidjua:latest
    ports:
      - "4200:4200"
    volumes:
      - sidjua-data:/data
    restart: unless-stopped

volumes:
  sidjua-data:
```

أو استخدم ملف `docker-compose.yml` المضمَّن الذي يُضيف وحدات تخزين مُسمَّاة للتكوين
والسجلات ومساحة عمل الوكيل، بالإضافة إلى خدمة Qdrant اختيارية للبحث الدلالي:

```bash
docker compose up -d
docker compose exec sidjua sidjua init
docker compose exec sidjua sidjua apply
```

## مزودو الخدمة

يتصل SIDJUA بأي مزود LLM بدون ربط:

| المزود | النماذج | مفتاح API |
|--------|---------|-----------|
| Anthropic | Claude Sonnet 4.6، Haiku 4.5، Opus 4.6 | `ANTHROPIC_API_KEY` |
| Google | Gemini 2.0، Gemini 1.5 Pro | `GOOGLE_API_KEY` |
| Groq | Llama 3.x، Mixtral | `GROQ_API_KEY` (المستوى المجاني) |
| Cloudflare Workers AI | Llama، Mistral، Phi | `CF_ACCOUNT_ID` + `CF_TOKEN` |
| Ollama | أي نموذج محلي | بدون مفتاح (محلي) |
| متوافق مع OpenAI | أي نقطة نهاية | URL مخصص + مفتاح |

```bash
# إضافة مفتاح مزود
sidjua key set groq gsk_...

# سرد المزودين والنماذج المتاحة
sidjua provider list
```
