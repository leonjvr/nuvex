## Встановлення

### Вимоги

| Інструмент | Потрібен | Примітки |
|------------|----------|----------|
| **Node.js** | >= 22.0.0 | Модулі ES, `fetch()`, `crypto.subtle`. [Завантажити](https://nodejs.org) |
| **Інструментарій C/C++** | Лише для збірки з джерел | `better-sqlite3` і `argon2` компілюють нативні доповнення |
| **Docker** | >= 24 (опціонально) | Лише для розгортання Docker |

Встановлення Node.js 22: Ubuntu/Debian (`curl -fsSL https://deb.nodesource.com/setup_22.x | sudo -E bash - && sudo apt-get install -y nodejs`), macOS (`brew install node@22`), Windows (`winget install OpenJS.NodeJS.LTS`).

Встановлення інструментів C/C++: Ubuntu (`sudo apt-get install -y python3 make g++ build-essential`), macOS (`xcode-select --install`), Windows (`npm install --global windows-build-tools`).

### Варіант A — Docker (Рекомендовано)

```bash
git clone https://github.com/GoetzKohlberg/sidjua.git
cd sidjua
docker compose up -d
```

```bash
# Переглянути автоматично згенерований API-ключ
docker compose exec sidjua cat /app/.system/api-key

# Ініціалізувати управління
docker compose exec sidjua sidjua apply --verbose

# Перевірка стану системи
docker compose exec sidjua sidjua selftest
```

Підтримує **linux/amd64** і **linux/arm64** (Raspberry Pi, Apple Silicon).

### Варіант Б — Глобальне встановлення npm

```bash
npm install -g sidjua
sidjua init          # Інтерактивне налаштування в 3 кроки
sidjua chat guide    # Керівництво ШІ без налаштування (без API-ключа)
```

### Варіант В — Збірка з джерел

```bash
git clone https://github.com/GoetzKohlberg/sidjua.git
cd sidjua
npm ci && npm run build
node dist/index.js --help
```

## Чому SIDJUA?

Кожен фреймворк агентів ШІ сьогодні спирається на те саме хибне припущення: що ви
можете довіряти ШІ дотриманню власних правил.

**Проблема управління на основі підказок:**

Ви надаєте агенту системну підказку «ніколи не звертайся до персональних даних клієнтів». Агент
читає інструкцію. Агент також читає повідомлення користувача з проханням отримати
історію платежів Івана Петренка. Агент вирішує — самостійно — чи виконувати команду.
Це не управління. Це наполегливо сформульована пропозиція.

**SIDJUA відрізняється.**

Управління знаходиться **поза** агентом. Кожна дія проходить через 5-етапний
конвеєр примусового виконання **до** виконання. Ви визначаєте правила в
YAML. Система їх виконує. Агент ніколи не вирішує, чи дотримуватись їх,
тому що перевірка відбувається до того, як агент діє.

Це управління через архітектуру — не через підказки, не через тонке налаштування,
не через надію.

## Як це працює

SIDJUA обгортає ваших агентів зовнішнім шаром управління. Виклик LLM агента
ніколи не відбувається, поки запропонована дія не пройде через 5-етапний конвеєр примусового виконання:

**Етап 1 — Заборонено:** Заблоковані дії відхиляються негайно. Немає виклику LLM,
немає запису в журналі, позначеного «дозволено», немає другого шансу. Якщо дія є у
списку заборонених, вона зупиняється тут.

**Етап 2 — Схвалення:** Дії, що потребують схвалення людини, затримуються для
схвалення перед виконанням. Агент чекає. Людина вирішує.

**Етап 3 — Бюджет:** Кожне завдання виконується з урахуванням лімітів вартості в реальному часі. Бюджети
на завдання і на агента застосовуються. Коли ліміт досягнуто, завдання
скасовується — не позначається, не реєструється для перегляду, *скасовується*.

**Етап 4 — Класифікація:** Дані, що перетинають межі підрозділів, перевіряються
за правилами класифікації. Агент Tier-2 не може отримати доступ до даних SECRET. Агент
у Підрозділі А не може читати секрети Підрозділу Б.

**Етап 5 — Політика:** Власні організаційні правила, що структурно виконуються. Обмеження
частоти викликів API, обмеження токенів виводу, обмеження часових вікон.

Весь конвеєр запускається до того, як будь-яка дія виконується. Не існує режиму «реєструй і
переглядай пізніше» для критично важливих для управління операцій.

### Єдиний файл конфігурації

Уся організація агентів знаходиться в одному файлі `divisions.yaml`:

```yaml
divisions:
  - name: engineering
    agents:
      - name: research-agent
        provider: anthropic
        model: claude-haiku-4-5-20251001
        tier: 2
        budget:
          per_task_usd: 0.50
          per_month_usd: 50.00
    governance:
      rules:
        - no_external_api_calls: true
        - max_tokens_per_response: 4096
        - require_human_approval: [delete, send_email]
```

`sidjua apply` читає цей файл і підготовлює повну інфраструктуру агентів:
агентів, підрозділи, RBAC, маршрутизацію, таблиці аудиту, шляхи до секретів і правила
управління — у 10 відтворюваних кроків.

### Архітектура агентів

Агенти організовані у **підрозділи** (функціональні групи) і **рівні**
(рівні довіри). Агенти Tier 1 мають повну автономію в межах своєї управляючої оболонки.
Агенти Tier 2 потребують схвалення для чутливих операцій. Агенти Tier 3
повністю під наглядом. Система рівнів виконується структурно — агент
не може самостійно підвищити свій рівень.

```
┌─────────────────────────────────────────────────┐
│                 SIDJUA Platform                 │
│                                                 │
│  ┌─────────────────────────────────────────┐   │
│  │           Governance Layer              │   │
│  │  Forbidden → Approval → Budget →        │   │
│  │  Classification → Policy (Stage 0)      │   │
│  └────────────────────┬────────────────────┘   │
│                       │ ✅ cleared              │
│            ┌──────────▼──────────┐             │
│            │   Agent Runtime     │             │
│            │  (any LLM provider) │             │
│            └──────────┬──────────┘             │
│                       │                        │
│  ┌────────────────────▼────────────────────┐   │
│  │            Audit Trail                  │   │
│  │  (WAL-integrity-verified, append-only)  │   │
│  └─────────────────────────────────────────┘   │
└─────────────────────────────────────────────────┘
```

## Архітектурні обмеження

SIDJUA застосовує ці обмеження на рівні архітектури — вони не можуть бути
вимкнені, обійдені або замінені агентами:

1. **Управління є зовнішнім**: Шар управління обгортає агента. Агент
   не має доступу до коду управління, не може змінювати правила і не може виявити,
   чи присутнє управління.

2. **До дії, не після дії**: Кожна дія перевіряється ДО виконання.
   Не існує режиму «реєструй і переглядай пізніше» для критично важливих для управління операцій.

3. **Структурне виконання**: Правила виконуються шляхами коду, а не
   підказками або інструкціями моделі. Агент не може «вирватися» із системи управління,
   тому що управління не реалізоване як інструкції для моделі.

4. **Незмінність аудиту**: Write-Ahead Log (WAL) є лише для додавання з
   перевіркою цілісності. Змінені записи виявляються і виключаються.

5. **Ізоляція підрозділів**: Агенти в різних підрозділах не можуть отримувати доступ
   до даних, секретів або каналів зв'язку один одного.

## Можливості

### Управління та відповідність

**Конвеєр до дії (Етап 0)** запускається перед кожною дією агента: перевірка заборонених
→ схвалення людини → виконання бюджету → класифікація даних → власна
політика. Всі п'ять етапів є структурними — вони виконуються в коді, а не в
підказці агента.

**Обов'язкові базові правила** постачаються з кожним встановленням: 10 правил управління
(`SYS-SEC-001` по `SYS-GOV-002`), які не можуть бути видалені або послаблені
користувацькою конфігурацією. Власні правила розширюють базу; вони не можуть її замінити.

**Відповідність EU AI Act** — журнал аудиту, система класифікації та робочі процеси
схвалення безпосередньо відповідають вимогам Статей 9, 12 і 17. Кінцевий термін
відповідності у серпні 2026 року вбудований у дорожню карту продукту.

**Звітність щодо відповідності** через `sidjua audit report/violations/agents/export`:
оцінка відповідності, оцінки довіри для кожного агента, історія порушень, експорт CSV/JSON
для зовнішніх аудиторів або інтеграції SIEM.

**Write-Ahead Log (WAL)** з перевіркою цілісності: кожне рішення управління
записується до журналу лише для додавання перед виконанням. Змінені записи
виявляються при читанні. `sidjua memory recover` повторно перевіряє та відновлює.

### Комунікації

Агенти не просто відповідають на виклики API — вони беруть участь у реальних каналах зв'язку.

**Двонаправлена електронна пошта** (`sidjua email status/test/threads`): агенти отримують
електронні листи через опитування IMAP і відповідають через SMTP. Зіставлення потоків через заголовки
In-Reply-To зберігає зв'язність розмов. Білий список відправників, обмеження розміру тіла
та видалення HTML захищають конвеєр агента від шкідливих вхідних даних.

**Бот-шлюз Discord**: повний інтерфейс slash-команд через `sidjua module install
discord`. Агенти відповідають на повідомлення Discord, підтримують потоки розмов
і надсилають проактивні сповіщення.

**Інтеграція з Telegram**: сповіщення агентів і повідомлення через бота Telegram.
Патерн адаптера з кількома каналами підтримує Telegram, Discord, ntfy і електронну пошту
паралельно.

### Операції

**Одна команда Docker** для виробництва:

```bash
docker run -p 4200:4200 ghcr.io/goetzkohlberg/sidjua:latest
```

API-ключ генерується автоматично при першому запуску і виводиться в журнали контейнера.
Змінні середовища не потрібні. Налаштування не потрібне. Сервер бази даних
не потрібний — SIDJUA використовує SQLite, один файл бази даних на агента.

**Управління через CLI** — повний життєвий цикл з одного бінарного файлу:

```bash
sidjua init                      # Інтерактивне налаштування робочого простору (3 кроки)
sidjua apply                     # Підготовка з divisions.yaml
sidjua agent create/list/stop    # Життєвий цикл агента
sidjua run "task..." --wait      # Надіслати завдання з виконанням управління
sidjua audit report              # Звіт про відповідність
sidjua costs                     # Розбивка витрат за підрозділом/агентом
sidjua backup create/restore     # Управління резервними копіями з підписом HMAC
sidjua update                    # Оновлення версії з автоматичним попереднім резервним копіюванням
sidjua rollback                  # Відновлення до попередньої версії одним кліком
sidjua email status/test         # Управління поштовим каналом
sidjua secret set/get/rotate     # Управління зашифрованими секретами
sidjua memory import/search      # Семантичний конвеєр знань
sidjua selftest                  # Перевірка стану системи (7 категорій, оцінка 0-100)
```

**Семантична пам'ять** — імпортуйте розмови та документи (`sidjua memory import
~/exports/claude-chats.zip`), шукайте з гібридним ранжуванням вектор + BM25. Підтримує
embeddings Cloudflare Workers AI (безкоштовно, без налаштування) і великі embeddings OpenAI
(вища якість для великих баз знань).

**Адаптивне розбиття** — конвеєр пам'яті автоматично підлаштовує розміри чанків, щоб залишатися
в межах ліміту токенів кожної моделі embedding.

**Керівництво без налаштування** — `sidjua chat guide` запускає інтерактивного AI-асистента
без будь-якого API-ключа, що працює на Cloudflare Workers AI через проксі SIDJUA.
Запитайте його, як налаштувати агентів, сконфігурувати управління або зрозуміти, що сталося
в журналі аудиту.

**Автономне розгортання** — працюйте повністю без підключення до інтернету, використовуючи локальні
LLM через Ollama або будь-яку кінцеву точку, сумісну з OpenAI. За замовчуванням немає телеметрії.
Опціональне звітування про помилки з повною редакцією PII.

### Безпека

**Ізоляція пісочниці** — навички агента виконуються всередині ізоляції процесів на рівні ОС через
bubblewrap (простори імен користувачів Linux). Нульові додаткові накладні витрати RAM. Підключаємий
інтерфейс `SandboxProvider`: `none` для розробки, `bubblewrap` для виробництва.

**Управління секретами** — зашифроване сховище секретів з RBAC (`sidjua secret
set/get/list/delete/rotate/namespaces`). Зовнішнє сховище не потрібне.

**Побудова з пріоритетом безпеки** — широкий внутрішній набір тестів плюс незалежна
перевірка двома зовнішніми аудиторами коду (DeepSeek V3 і xAI Grok). Заголовки
безпеки, захист від CSRF, обмеження швидкості та санітизація вхідних даних на кожній поверхні API.
Запобігання SQL-ін'єкціям з параметризованими запитами всюди.

**Цілісність резервних копій** — архіви резервних копій з підписом HMAC із захистом від zip-slip,
запобіганням zip-бомбам і перевіркою контрольної суми маніфесту при відновленні.

## Імпорт з інших фреймворків

```bash
# Попередній перегляд того, що буде імпортовано — без змін
sidjua import openclaw --dry-run

# Імпортувати конфігурацію + файли навичок
sidjua import openclaw --skills
```

Ваші існуючі агенти зберігають свою ідентичність, моделі та навички. SIDJUA автоматично додає
управління, журнали аудиту та контроль бюджету.

## Довідник з конфігурації

Мінімальний `divisions.yaml` для початку роботи:

```yaml
organization:
  name: "my-org"
  tier: 1

divisions:
  - name: operations
    tier: 2
    agents:
      - name: ops-agent
        provider: anthropic
        model: claude-haiku-4-5-20251001
        division: operations
        budget:
          per_task_usd: 0.25
          per_month_usd: 25.00

governance:
  stage0:
    enabled: true
    forbidden_actions:
      - delete_database
      - exfiltrate_data
    classification:
      default_level: INTERNAL
      max_agent_level: CONFIDENTIAL
```

`sidjua apply` підготовлює повну інфраструктуру з цього файлу. Запустіть повторно
після змін — команда є ідемпотентною.

Дивіться [docs/SIDJUA-APPLY-TECH-SPEC-V1.md](docs/SIDJUA-APPLY-TECH-SPEC-V1.md)
для повної специфікації всіх 10 кроків підготовки.

## REST API

SIDJUA REST API працює на тому самому порту, що й панель управління:

```bash
sidjua server start --port 4200 --api-key $SIDJUA_API_KEY
```

Ключові кінцеві точки:

```
GET  /api/v1/health          # Публічна перевірка стану (без автентифікації)
GET  /api/v1/info            # Метадані системи (автентифіковано)
POST /api/v1/execute/run     # Надіслати завдання
GET  /api/v1/execute/:id/status  # Статус завдання
GET  /api/v1/execute/:id/result  # Результат завдання
GET  /api/v1/events          # Потік подій SSE
GET  /api/v1/audit/report    # Звіт про відповідність
```

Усі кінцеві точки, крім `/health`, потребують автентифікації Bearer. Створіть ключ:

```bash
sidjua api-key generate
```

## Docker Compose

```yaml
services:
  sidjua:
    image: ghcr.io/goetzkohlberg/sidjua:latest
    ports:
      - "4200:4200"
    volumes:
      - sidjua-data:/data
    restart: unless-stopped

volumes:
  sidjua-data:
```

Або використовуйте доданий `docker-compose.yml`, який додає іменовані томи для конфігурації,
журналів і робочого простору агента, плюс опціональний сервіс Qdrant для семантичного пошуку:

```bash
docker compose up -d
docker compose exec sidjua sidjua init
docker compose exec sidjua sidjua apply
```

## Провайдери

SIDJUA підключається до будь-якого провайдера LLM без прив'язки до постачальника:

| Провайдер | Моделі | API-ключ |
|-----------|--------|----------|
| Anthropic | Claude Sonnet 4.6, Haiku 4.5, Opus 4.6 | `ANTHROPIC_API_KEY` |
| Google | Gemini 2.0, Gemini 1.5 Pro | `GOOGLE_API_KEY` |
| Groq | Llama 3.x, Mixtral | `GROQ_API_KEY` (безкоштовний рівень) |
| Cloudflare Workers AI | Llama, Mistral, Phi | `CF_ACCOUNT_ID` + `CF_TOKEN` |
| Ollama | Будь-яка локальна модель | Ключ не потрібен (локальний) |
| Сумісний з OpenAI | Будь-яка кінцева точка | Власний URL + ключ |

```bash
# Додати ключ провайдера
sidjua key set groq gsk_...

# Список доступних провайдерів і моделей
sidjua provider list
```

| Близький Схід | Арабська |
| Азія | Гінді, Бенгальська, Філіппінська, Індонезійська, Малайська, Тайська, В'єтнамська, Японська, Корейська, Китайська (спрощена), Китайська (традиційна) |

- Правильним перекладом

Хочете підтримувати мову? Дивіться [CONTRIBUTING.md](CONTRIBUTING.md#translations) — ми використовуємо модель обслуговування для кожної мови.
