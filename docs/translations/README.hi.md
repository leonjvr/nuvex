## इंस्टॉलेशन

### पूर्वापेक्षाएँ

| टूल | आवश्यक | नोट्स |
|-----|--------|-------|
| **Node.js** | >= 22.0.0 | ES मॉड्यूल, `fetch()`, `crypto.subtle`। [डाउनलोड करें](https://nodejs.org) |
| **C/C++ टूलचेन** | केवल सोर्स बिल्ड के लिए | `better-sqlite3` और `argon2` नेटिव ऐड-ऑन कंपाइल करते हैं |
| **Docker** | >= 24 (वैकल्पिक) | केवल Docker डिप्लॉयमेंट के लिए |

Node.js 22 इंस्टॉल करें: Ubuntu/Debian (`curl -fsSL https://deb.nodesource.com/setup_22.x | sudo -E bash - && sudo apt-get install -y nodejs`), macOS (`brew install node@22`), Windows (`winget install OpenJS.NodeJS.LTS`)।

C/C++ टूल्स इंस्टॉल करें: Ubuntu (`sudo apt-get install -y python3 make g++ build-essential`), macOS (`xcode-select --install`), Windows (`npm install --global windows-build-tools`)।

### विकल्प A — Docker (अनुशंसित)

```bash
git clone https://github.com/GoetzKohlberg/sidjua.git
cd sidjua
docker compose up -d
```

```bash
# स्वचालित रूप से जेनरेट हुई API key देखें
docker compose exec sidjua cat /app/.system/api-key

# गवर्नेंस बूटस्ट्रैप करें
docker compose exec sidjua sidjua apply --verbose

# सिस्टम हेल्थ चेक
docker compose exec sidjua sidjua selftest
```

**linux/amd64** और **linux/arm64** (Raspberry Pi, Apple Silicon) का समर्थन करता है।

### विकल्प B — npm ग्लोबल इंस्टॉल

```bash
npm install -g sidjua
sidjua init          # इंटरएक्टिव 3-चरण सेटअप
sidjua chat guide    # जीरो-कॉन्फिग AI गाइड (कोई API key नहीं चाहिए)
```

### विकल्प C — सोर्स बिल्ड

```bash
git clone https://github.com/GoetzKohlberg/sidjua.git
cd sidjua
npm ci && npm run build
node dist/index.js --help
```

## SIDJUA क्यों?

आज का हर AI एजेंट फ्रेमवर्क एक ही टूटे हुए अनुमान पर निर्भर करता है: कि आप
AI पर भरोसा कर सकते हैं कि वह अपने नियमों का पालन करेगा।

**प्रॉम्प्ट-आधारित गवर्नेंस की समस्या:**

आप एक एजेंट को एक सिस्टम प्रॉम्प्ट देते हैं जो कहता है "ग्राहक के PII को कभी एक्सेस मत करो।"
एजेंट निर्देश पढ़ता है। एजेंट उपयोगकर्ता का संदेश भी पढ़ता है जिसमें जॉन स्मिथ का
भुगतान इतिहास निकालने को कहा गया है। एजेंट — खुद से — तय करता है कि पालन करना है या नहीं।
यह गवर्नेंस नहीं है। यह एक दृढ़ सुझाव है।

**SIDJUA अलग है।**

गवर्नेंस एजेंट के **बाहर** बैठती है। हर कार्य निष्पादन से **पहले** एक 5-चरण
प्री-एक्शन एनफोर्समेंट पाइपलाइन से गुज़रता है। आप YAML में नियम परिभाषित करते हैं।
सिस्टम उन्हें लागू करता है। एजेंट को कभी यह तय करने का मौका नहीं मिलता कि उनका
पालन करना है या नहीं, क्योंकि जाँच एजेंट के कार्य करने से पहले होती है।

यह आर्किटेक्चर द्वारा गवर्नेंस है — प्रॉम्प्टिंग से नहीं, फाइन-ट्यूनिंग से नहीं,
उम्मीद से नहीं।

## यह कैसे काम करता है

SIDJUA आपके एजेंटों को एक बाहरी गवर्नेंस परत में लपेटता है। एजेंट का LLM
कॉल तब तक नहीं होता जब तक प्रस्तावित कार्य 5-चरण एनफोर्समेंट पाइपलाइन को पार नहीं कर लेता:

**चरण 1 — प्रतिबंधित:** अवरुद्ध कार्य तुरंत अस्वीकार कर दिए जाते हैं। कोई LLM
कॉल नहीं, कोई लॉग एंट्री "अनुमत" चिह्नित नहीं, कोई दूसरा मौका नहीं। यदि कार्य
प्रतिबंधित सूची में है, तो यह यहीं रुकता है।

**चरण 2 — अनुमोदन:** जिन कार्यों के लिए मानव हस्ताक्षर की आवश्यकता होती है, उन्हें
निष्पादन से पहले अनुमोदन के लिए रोका जाता है। एजेंट प्रतीक्षा करता है। मानव निर्णय लेता है।

**चरण 3 — बजट:** हर टास्क रियल-टाइम लागत सीमाओं के विरुद्ध चलता है। प्रति-टास्क
और प्रति-एजेंट बजट लागू होते हैं। सीमा पहुँचने पर, टास्क रद्द हो जाता है — फ्लैग नहीं
किया जाता, समीक्षा के लिए लॉग नहीं किया जाता, *रद्द* हो जाता है।

**चरण 4 — वर्गीकरण:** विभाग की सीमाओं को पार करने वाला डेटा वर्गीकरण नियमों के
विरुद्ध जाँचा जाता है। एक Tier-2 एजेंट SECRET डेटा एक्सेस नहीं कर सकता। डिवीजन A
का एजेंट डिवीजन B के रहस्य नहीं पढ़ सकता।

**चरण 5 — नीति:** कस्टम संगठनात्मक नियम, संरचनात्मक रूप से लागू। API कॉल
आवृत्ति सीमाएँ, आउटपुट टोकन कैप्स, टाइम-विंडो प्रतिबंध।

पूरी पाइपलाइन किसी भी कार्य के निष्पादन से पहले चलती है। गवर्नेंस-महत्वपूर्ण
ऑपरेशनों के लिए कोई "लॉग करें और बाद में समीक्षा करें" मोड नहीं है।

### एकल कॉन्फ़िगरेशन फ़ाइल

आपका पूरा एजेंट संगठन एक `divisions.yaml` में रहता है:

```yaml
divisions:
  - name: engineering
    agents:
      - name: research-agent
        provider: anthropic
        model: claude-haiku-4-5-20251001
        tier: 2
        budget:
          per_task_usd: 0.50
          per_month_usd: 50.00
    governance:
      rules:
        - no_external_api_calls: true
        - max_tokens_per_response: 4096
        - require_human_approval: [delete, send_email]
```

`sidjua apply` इस फ़ाइल को पढ़ता है और पूर्ण एजेंट इंफ्रास्ट्रक्चर प्रोविज़न करता है:
एजेंट, डिवीजन, RBAC, रूटिंग, ऑडिट टेबल, सीक्रेट पाथ, और गवर्नेंस
नियम — 10 पुनरुत्पादन योग्य चरणों में।

### एजेंट आर्किटेक्चर

एजेंट **डिवीजनों** (कार्यात्मक समूहों) और **टियर्स** (विश्वास स्तरों) में
व्यवस्थित होते हैं। टियर 1 एजेंटों को उनके गवर्नेंस लिफाफे के भीतर पूर्ण स्वायत्तता है।
टियर 2 एजेंटों को संवेदनशील ऑपरेशनों के लिए अनुमोदन की आवश्यकता होती है। टियर 3
एजेंट पूरी तरह से निगरानी में हैं। टियर सिस्टम संरचनात्मक रूप से लागू होता है —
एक एजेंट खुद को पदोन्नत नहीं कर सकता।

## आर्किटेक्चर बाधाएँ

SIDJUA इन बाधाओं को आर्किटेक्चर स्तर पर लागू करता है — इन्हें एजेंटों द्वारा
अक्षम, बाईपास या ओवरराइड नहीं किया जा सकता:

1. **गवर्नेंस बाहरी है**: गवर्नेंस परत एजेंट को लपेटती है। एजेंट के पास गवर्नेंस
   कोड तक कोई पहुँच नहीं है, नियम नहीं बदल सकता, और यह पता नहीं कर सकता कि
   गवर्नेंस मौजूद है या नहीं।

2. **प्री-एक्शन, पोस्ट-एक्शन नहीं**: हर कार्य निष्पादन से पहले जाँचा जाता है।
   गवर्नेंस-महत्वपूर्ण ऑपरेशनों के लिए कोई "लॉग करें और बाद में समीक्षा करें" मोड नहीं है।

3. **संरचनात्मक प्रवर्तन**: नियम कोड पाथ द्वारा लागू होते हैं, प्रॉम्प्ट या मॉडल
   निर्देशों द्वारा नहीं। एक एजेंट गवर्नेंस से "जेलब्रेक" नहीं कर सकता क्योंकि
   गवर्नेंस मॉडल के लिए निर्देशों के रूप में लागू नहीं है।

4. **ऑडिट अपरिवर्तनीयता**: राइट-अहेड लॉग (WAL) अखंडता सत्यापन के साथ केवल
   अपेंड-ओनली है। छेड़छाड़ की गई प्रविष्टियाँ पढ़ने पर पता चल जाती हैं और बाहर कर दी जाती हैं।

5. **डिवीजन आइसोलेशन**: अलग-अलग डिवीजनों के एजेंट एक-दूसरे के डेटा, रहस्यों
   या संचार चैनलों तक नहीं पहुँच सकते।

## फ़ीचर्स

### गवर्नेंस और अनुपालन

**प्री-एक्शन पाइपलाइन (स्टेज 0)** हर एजेंट एक्शन से पहले चलती है: प्रतिबंधित
जाँच → मानव अनुमोदन → बजट प्रवर्तन → डेटा वर्गीकरण → कस्टम नीति।
पाँचों चरण संरचनात्मक हैं — वे कोड में निष्पादित होते हैं, एजेंट के प्रॉम्प्ट में नहीं।

**अनिवार्य बेसलाइन नियम** हर इंस्टॉलेशन के साथ आते हैं: 10 गवर्नेंस नियम
(`SYS-SEC-001` से `SYS-GOV-002` तक) जिन्हें उपयोगकर्ता कॉन्फ़िगरेशन द्वारा हटाया
या कमज़ोर नहीं किया जा सकता। कस्टम नियम बेसलाइन का विस्तार करते हैं; वे इसे ओवरराइड नहीं कर सकते।

**EU AI Act अनुपालन** — ऑडिट ट्रेल, वर्गीकरण ढाँचा, और अनुमोदन
वर्कफ़्लो सीधे अनुच्छेद 9, 12 और 17 की आवश्यकताओं से मेल खाते हैं। अगस्त 2026
की अनुपालन समयसीमा उत्पाद रोडमैप में बनी हुई है।

**अनुपालन रिपोर्टिंग** `sidjua audit report/violations/agents/export` के माध्यम से:
अनुपालन स्कोर, प्रति-एजेंट ट्रस्ट स्कोर, उल्लंघन इतिहास, बाहरी ऑडिटर या SIEM
एकीकरण के लिए CSV/JSON निर्यात।

**राइट-अहेड लॉग (WAL)** अखंडता सत्यापन के साथ: हर गवर्नेंस निर्णय निष्पादन
से पहले एक अपेंड-ओनली लॉग में लिखा जाता है। छेड़छाड़ की गई प्रविष्टियाँ पढ़ते समय
पता चल जाती हैं। `sidjua memory recover` पुनः सत्यापन और मरम्मत करता है।

### संचार

एजेंट सिर्फ API कॉल का जवाब नहीं देते — वे वास्तविक संचार
चैनलों में भाग लेते हैं।

**द्विदिशात्मक ईमेल** (`sidjua email status/test/threads`): एजेंट IMAP पोलिंग के
माध्यम से ईमेल प्राप्त करते हैं और SMTP के माध्यम से उत्तर देते हैं। In-Reply-To
हेडर के माध्यम से थ्रेड मैपिंग बातचीत को सुसंगत रखती है। प्रेषक व्हाइटलिस्टिंग,
बॉडी साइज़ सीमाएँ, और HTML स्ट्रिपिंग एजेंट पाइपलाइन को दुर्भावनापूर्ण इनपुट से
बचाती है।

**Discord गेटवे बॉट**: `sidjua module install discord` के माध्यम से पूर्ण
स्लैश-कमांड इंटरफेस। एजेंट Discord संदेशों का जवाब देते हैं, बातचीत के थ्रेड
बनाए रखते हैं, और सक्रिय सूचनाएँ भेजते हैं।

**Telegram एकीकरण**: Telegram बॉट के माध्यम से एजेंट अलर्ट और सूचनाएँ।
मल्टी-चैनल एडाप्टर पैटर्न Telegram, Discord, ntfy, और ईमेल को समानांतर में समर्थन करता है।

### ऑपरेशंस

उत्पादन के लिए **एकल Docker कमांड**:

```bash
docker run -p 4200:4200 ghcr.io/goetzkohlberg/sidjua:latest
```

API key पहली बार शुरू होने पर स्वचालित रूप से जेनरेट होती है और कंटेनर लॉग में
प्रिंट होती है। कोई एनवायरनमेंट वेरिएबल आवश्यक नहीं। कोई कॉन्फ़िगरेशन आवश्यक नहीं।
कोई डेटाबेस सर्वर आवश्यक नहीं — SIDJUA SQLite का उपयोग करता है, प्रति एजेंट एक
डेटाबेस फ़ाइल।

**CLI प्रबंधन** — एकल बाइनरी से पूर्ण जीवन-चक्र:

```bash
sidjua init                      # इंटरेक्टिव वर्कस्पेस सेटअप (3 चरण)
sidjua apply                     # divisions.yaml से प्रोविज़न करें
sidjua agent create/list/stop    # एजेंट जीवन-चक्र
sidjua run "task..." --wait      # गवर्नेंस प्रवर्तन के साथ टास्क सबमिट करें
sidjua audit report              # अनुपालन रिपोर्ट
sidjua costs                     # डिवीजन/एजेंट द्वारा लागत विभाजन
sidjua backup create/restore     # HMAC-हस्ताक्षरित बैकअप प्रबंधन
sidjua update                    # स्वचालित प्री-बैकअप के साथ संस्करण अपडेट
sidjua rollback                  # पिछले संस्करण पर 1-क्लिक पुनर्स्थापना
sidjua email status/test         # ईमेल चैनल प्रबंधन
sidjua secret set/get/rotate     # एन्क्रिप्टेड रहस्य प्रबंधन
sidjua memory import/search      # सिमेंटिक नॉलेज पाइपलाइन
sidjua selftest                  # सिस्टम हेल्थ चेक (7 श्रेणियाँ, 0-100 स्कोर)
```

**सिमेंटिक मेमरी** — बातचीत और दस्तावेज़ आयात करें (`sidjua memory import
~/exports/claude-chats.zip`), वेक्टर + BM25 हाइब्रिड रैंकिंग के साथ खोजें।
Cloudflare Workers AI एम्बेडिंग (मुफ़्त, जीरो-कॉन्फिग) और OpenAI बड़े एम्बेडिंग
(बड़े नॉलेज बेस के लिए उच्च गुणवत्ता) का समर्थन करता है।

**अनुकूली चंकिंग** — मेमरी पाइपलाइन स्वचालित रूप से प्रत्येक एम्बेडिंग मॉडल की
टोकन सीमा के भीतर रहने के लिए चंक आकार समायोजित करती है।

**जीरो-कॉन्फिग गाइड** — `sidjua chat guide` बिना किसी API key के एक इंटरेक्टिव AI
असिस्टेंट लॉन्च करता है, SIDJUA प्रॉक्सी के माध्यम से Cloudflare Workers AI द्वारा
संचालित। एजेंट सेटअप करने, गवर्नेंस कॉन्फ़िगर करने, या ऑडिट लॉग में क्या हुआ
समझने के लिए पूछें।

**एयर-गैप डिप्लॉयमेंट** — Ollama या किसी भी OpenAI-संगत एंडपॉइंट के माध्यम से
स्थानीय LLM का उपयोग करके इंटरनेट से पूरी तरह से डिस्कनेक्टेड चलाएँ। डिफ़ॉल्ट
रूप से कोई टेलीमेट्री नहीं। पूर्ण PII रिडैक्शन के साथ वैकल्पिक ऑप्ट-इन क्रैश रिपोर्टिंग।

### सुरक्षा

**सैंडबॉक्स आइसोलेशन** — एजेंट स्किल्स bubblewrap (Linux यूजर नेमस्पेस) के
माध्यम से OS-स्तरीय प्रक्रिया आइसोलेशन के अंदर चलती हैं। शून्य अतिरिक्त RAM
ओवरहेड। प्लगेबल `SandboxProvider` इंटरफेस: विकास के लिए `none`, उत्पादन के लिए `bubblewrap`।

**रहस्य प्रबंधन** — RBAC के साथ एन्क्रिप्टेड रहस्य स्टोर (`sidjua secret
set/get/list/delete/rotate/namespaces`)। कोई बाहरी वॉल्ट आवश्यक नहीं।

**सुरक्षा-पहले बिल्ड** — व्यापक आंतरिक टेस्ट सूट साथ ही 2 बाहरी कोड ऑडिटर
(DeepSeek V3 और xAI Grok) द्वारा स्वतंत्र सत्यापन। हर API सरफेस पर सुरक्षा
हेडर, CSRF सुरक्षा, रेट लिमिटिंग, और इनपुट सैनिटाइज़ेशन। पूरे सिस्टम में
पैरामीटरयुक्त क्वेरी के साथ SQL इंजेक्शन रोकथाम।

**बैकअप अखंडता** — HMAC-हस्ताक्षरित बैकअप आर्काइव zip-slip सुरक्षा,
zip बम रोकथाम, और पुनर्स्थापना पर मैनिफेस्ट चेकसम सत्यापन के साथ।

## कॉन्फ़िगरेशन संदर्भ

शुरुआत के लिए एक न्यूनतम `divisions.yaml`:

```yaml
organization:
  name: "my-org"
  tier: 1

divisions:
  - name: operations
    tier: 2
    agents:
      - name: ops-agent
        provider: anthropic
        model: claude-haiku-4-5-20251001
        division: operations
        budget:
          per_task_usd: 0.25
          per_month_usd: 25.00

governance:
  stage0:
    enabled: true
    forbidden_actions:
      - delete_database
      - exfiltrate_data
    classification:
      default_level: INTERNAL
      max_agent_level: CONFIDENTIAL
```

`sidjua apply` इस फ़ाइल से पूर्ण इंफ्रास्ट्रक्चर प्रोविज़न करता है। बदलावों
के बाद फिर चलाएँ — यह इडेम्पोटेंट है।

सभी 10 प्रोविज़निंग चरणों की पूर्ण विशिष्टता के लिए
[docs/SIDJUA-APPLY-TECH-SPEC-V1.md](docs/SIDJUA-APPLY-TECH-SPEC-V1.md) देखें।

## REST API

SIDJUA REST API डैशबोर्ड के समान पोर्ट पर चलती है:

```bash
sidjua server start --port 4200 --api-key $SIDJUA_API_KEY
```

मुख्य एंडपॉइंट:

```
GET  /api/v1/health          # सार्वजनिक हेल्थ चेक (कोई auth नहीं)
GET  /api/v1/info            # सिस्टम मेटाडेटा (प्रमाणित)
POST /api/v1/execute/run     # टास्क सबमिट करें
GET  /api/v1/execute/:id/status  # टास्क स्थिति
GET  /api/v1/execute/:id/result  # टास्क परिणाम
GET  /api/v1/events          # SSE इवेंट स्ट्रीम
GET  /api/v1/audit/report    # अनुपालन रिपोर्ट
```

`/health` के अलावा सभी एंडपॉइंट Bearer प्रमाणीकरण की आवश्यकता है। एक key जेनरेट करें:

```bash
sidjua api-key generate
```

## Docker Compose

```yaml
services:
  sidjua:
    image: ghcr.io/goetzkohlberg/sidjua:latest
    ports:
      - "4200:4200"
    volumes:
      - sidjua-data:/data
    restart: unless-stopped

volumes:
  sidjua-data:
```

या शामिल `docker-compose.yml` का उपयोग करें जो config, लॉग और एजेंट
वर्कस्पेस के लिए नामित वॉल्यूम जोड़ता है, साथ ही सिमेंटिक सर्च के लिए
एक वैकल्पिक Qdrant सेवा:

```bash
docker compose up -d
docker compose exec sidjua sidjua init
docker compose exec sidjua sidjua apply
```

## प्रदाता

SIDJUA बिना किसी लॉक-इन के किसी भी LLM प्रदाता से जुड़ता है:

| प्रदाता | मॉडल | API कुंजी |
|---------|------|-----------|
| Anthropic | Claude Sonnet 4.6, Haiku 4.5, Opus 4.6 | `ANTHROPIC_API_KEY` |
| Google | Gemini 2.0, Gemini 1.5 Pro | `GOOGLE_API_KEY` |
| Groq | Llama 3.x, Mixtral | `GROQ_API_KEY` (मुफ़्त टियर) |
| Cloudflare Workers AI | Llama, Mistral, Phi | `CF_ACCOUNT_ID` + `CF_TOKEN` |
| Ollama | कोई भी स्थानीय मॉडल | कोई कुंजी नहीं (स्थानीय) |
| OpenAI-संगत | कोई भी एंडपॉइंट | कस्टम URL + कुंजी |

```bash
# एक प्रदाता key जोड़ें
sidjua key set groq gsk_...

# उपलब्ध प्रदाताओं और मॉडल सूचीबद्ध करें
sidjua provider list
```
