# SIDJUA — Платформа управления агентами ИИ

> Единственная платформа агентов, где управление обеспечивается архитектурой, а не надеждой на то, что модель будет вести себя правильно.

## Почему SIDJUA?

Каждый фреймворк агентов ИИ сегодня опирается на одно и то же ошибочное предположение: что вы
можете доверять ИИ соблюдению его собственных правил.

**Проблема управления на основе подсказок:**

Вы даёте агенту системную подсказку «никогда не обращайся к персональным данным клиентов». Агент
читает инструкцию. Агент также читает сообщение пользователя с просьбой извлечь историю платежей
Ивана Иванова. Агент решает — самостоятельно — подчиняться ли. Это не управление. Это
настойчиво сформулированное предложение.

**SIDJUA отличается.**

Управление находится **вне** агента. Каждое действие проходит через 5-этапный
конвейер принудительного исполнения **до** выполнения. Вы определяете правила в
YAML. Система их исполняет. Агент никогда не решает, следовать ли им,
потому что проверка происходит до того, как агент действует.

Это управление через архитектуру — не через подсказки, не через тонкую настройку,
не через надежду.

## Как это работает

SIDJUA оборачивает ваших агентов во внешний уровень управления. Вызов LLM агента
никогда не происходит, пока предложенное действие не пройдёт через 5-этапный конвейер принудительного исполнения:

**Этап 1 — Запрещено:** Заблокированные действия отклоняются немедленно. Нет вызова LLM,
нет записи в журнале, помеченной «разрешено», нет второго шанса. Если действие находится в
списке запрещённых, оно останавливается здесь.

**Этап 2 — Одобрение:** Действия, требующие одобрения человека, задерживаются для
одобрения перед выполнением. Агент ждёт. Человек решает.

**Этап 3 — Бюджет:** Каждое задание выполняется с учётом лимитов стоимости в реальном времени. Бюджеты
на задание и на агента соблюдаются. Когда лимит достигнут, задание
отменяется — не помечается, не регистрируется для проверки, *отменяется*.

**Этап 4 — Классификация:** Данные, пересекающие границы подразделений, проверяются
по правилам классификации. Агент Tier-2 не может получить доступ к данным SECRET. Агент
в Подразделении А не может читать секреты Подразделения Б.

**Этап 5 — Политика:** Пользовательские организационные правила, структурно соблюдаемые. Ограничения
частоты вызовов API, ограничения токенов вывода, ограничения временных окон.

Весь конвейер выполняется до того, как какое-либо действие будет выполнено. Для критически важных для управления
операций не существует режима «регистрировать и проверять позже».

### Единый файл конфигурации

Вся организация агентов находится в одном файле `divisions.yaml`:

```yaml
divisions:
  - name: engineering
    agents:
      - name: research-agent
        provider: anthropic
        model: claude-haiku-4-5-20251001
        tier: 2
        budget:
          per_task_usd: 0.50
          per_month_usd: 50.00
    governance:
      rules:
        - no_external_api_calls: true
        - max_tokens_per_response: 4096
        - require_human_approval: [delete, send_email]
```

`sidjua apply` читает этот файл и создаёт полную инфраструктуру агентов:
агентов, подразделения, RBAC, маршрутизацию, таблицы аудита, пути к секретам и правила
управления — в 10 воспроизводимых шагов.

### Архитектура агентов

Агенты организованы в **подразделения** (функциональные группы) и **уровни**
(уровни доверия). Агенты Tier 1 имеют полную автономию в рамках своей управляющей оболочки.
Агенты Tier 2 требуют одобрения для чувствительных операций. Агенты Tier 3
полностью находятся под надзором. Система уровней соблюдается структурно — агент
не может самостоятельно повысить свой уровень.

```
┌─────────────────────────────────────────────────┐
│                 SIDJUA Platform                 │
│                                                 │
│  ┌─────────────────────────────────────────┐   │
│  │           Governance Layer              │   │
│  │  Forbidden → Approval → Budget →        │   │
│  │  Classification → Policy (Stage 0)      │   │
│  └────────────────────┬────────────────────┘   │
│                       │ ✅ cleared              │
│            ┌──────────▼──────────┐             │
│            │   Agent Runtime     │             │
│            │  (any LLM provider) │             │
│            └──────────┬──────────┘             │
│                       │                        │
│  ┌────────────────────▼────────────────────┐   │
│  │            Audit Trail                  │   │
│  │  (WAL-integrity-verified, append-only)  │   │
│  └─────────────────────────────────────────┘   │
└─────────────────────────────────────────────────┘
```

## Архитектурные ограничения

SIDJUA соблюдает эти ограничения на уровне архитектуры — они не могут быть
отключены, обойдены или переопределены агентами:

1. **Управление является внешним**: Уровень управления оборачивает агента. Агент
   не имеет доступа к коду управления, не может изменять правила и не может обнаружить,
   присутствует ли управление.

2. **До действия, не после действия**: Каждое действие проверяется ДО выполнения.
   Для критически важных для управления операций не существует режима «регистрировать и проверять позже».

3. **Структурное принудительное исполнение**: Правила применяются через пути кода, а не через
   подсказки или инструкции модели. Агент не может «сбежать» из системы управления,
   потому что управление не реализовано как инструкции для модели.

4. **Неизменность аудита**: Write-Ahead Log (WAL) является только для добавления с
   проверкой целостности. Изменённые записи обнаруживаются и исключаются.

5. **Изоляция подразделений**: Агенты в разных подразделениях не могут получать доступ
   к данным, секретам или каналам связи друг друга.

## Возможности

### Управление и соответствие требованиям

**Конвейер до действия (Этап 0)** выполняется перед каждым действием агента: проверка запрещённых
→ одобрение человека → соблюдение бюджета → классификация данных → пользовательская
политика. Все пять этапов структурны — они выполняются в коде, а не в
подсказке агента.

**Обязательные базовые правила** поставляются с каждой установкой: 10 правил управления
(`SYS-SEC-001` по `SYS-GOV-002`), которые не могут быть удалены или ослаблены
пользовательской конфигурацией. Пользовательские правила расширяют базу; они не могут её переопределить.

**Соответствие EU AI Act** — журнал аудита, система классификации и рабочие процессы
одобрения напрямую соответствуют требованиям статей 9, 12 и 17. Крайний срок
соответствия в августе 2026 года встроен в дорожную карту продукта.

**Отчётность по соответствию** через `sidjua audit report/violations/agents/export`:
оценка соответствия, оценки доверия для каждого агента, история нарушений, экспорт CSV/JSON
для внешних аудиторов или интеграции SIEM.

**Write-Ahead Log (WAL)** с проверкой целостности: каждое решение управления
записывается в журнал только для добавления перед выполнением. Изменённые записи
обнаруживаются при чтении. `sidjua memory recover` повторно проверяет и восстанавливает.

### Коммуникации

Агенты не просто отвечают на вызовы API — они участвуют в реальных каналах коммуникации.

**Двунаправленная электронная почта** (`sidjua email status/test/threads`): агенты получают
электронные письма через опрос IMAP и отвечают через SMTP. Сопоставление потоков через заголовки
In-Reply-To сохраняет связность разговоров. Белый список отправителей, ограничения размера тела
и удаление HTML защищают конвейер агента от вредоносных входных данных.

**Бот-шлюз Discord**: полный интерфейс slash-команд через `sidjua module install
discord`. Агенты отвечают на сообщения Discord, поддерживают потоки разговоров
и отправляют проактивные уведомления.

**Интеграция с Telegram**: оповещения агентов и уведомления через бота Telegram.
Паттерн адаптера с несколькими каналами поддерживает Telegram, Discord, ntfy и электронную почту
параллельно.

### Операции

**Одна команда Docker** для производства:

```bash
docker run -p 4200:4200 ghcr.io/goetzkohlberg/sidjua:latest
```

API-ключ генерируется автоматически при первом запуске и выводится в журналы контейнера.
Переменные среды не требуются. Настройка не требуется. Сервер базы данных
не требуется — SIDJUA использует SQLite, один файл базы данных на агента.

**Управление через CLI** — полный жизненный цикл из одного бинарного файла:

```bash
sidjua init                      # Интерактивная настройка рабочего пространства (3 шага)
sidjua apply                     # Подготовка из divisions.yaml
sidjua agent create/list/stop    # Жизненный цикл агента
sidjua run "task..." --wait      # Отправить задание с соблюдением управления
sidjua audit report              # Отчёт о соответствии
sidjua costs                     # Разбивка затрат по подразделению/агенту
sidjua backup create/restore     # Управление резервными копиями с подписью HMAC
sidjua update                    # Обновление версии с автоматическим предварительным резервным копированием
sidjua rollback                  # Восстановление к предыдущей версии одним щелчком
sidjua email status/test         # Управление почтовым каналом
sidjua secret set/get/rotate     # Управление зашифрованными секретами
sidjua memory import/search      # Семантический конвейер знаний
sidjua selftest                  # Проверка состояния системы (7 категорий, оценка 0-100)
```

**Семантическая память** — импортируйте разговоры и документы (`sidjua memory import
~/exports/claude-chats.zip`), ищите с гибридным ранжированием вектор + BM25. Поддерживает
embeddings Cloudflare Workers AI (бесплатно, без настройки) и большие embeddings OpenAI
(более высокое качество для крупных баз знаний).

**Адаптивная нарезка** — конвейер памяти автоматически подстраивает размеры чанков, чтобы оставаться
в пределах лимита токенов каждой модели embedding.

**Руководство без настройки** — `sidjua chat guide` запускает интерактивного AI-ассистента
без какого-либо API-ключа, работающего на Cloudflare Workers AI через прокси SIDJUA.
Спросите его, как настроить агентов, сконфигурировать управление или понять, что произошло
в журнале аудита.

**Автономное развёртывание** — работайте полностью без подключения к интернету, используя локальные
LLM через Ollama или любую конечную точку, совместимую с OpenAI. По умолчанию нет телеметрии.
Опциональная отчётность об ошибках с полной редакцией PII.

### Безопасность

**Изоляция в песочнице** — навыки агента выполняются внутри изоляции процессов на уровне ОС через
bubblewrap (пространства имён пользователей Linux). Нулевые дополнительные накладные расходы RAM. Подключаемый
интерфейс `SandboxProvider`: `none` для разработки, `bubblewrap` для производства.

**Управление секретами** — зашифрованное хранилище секретов с RBAC (`sidjua secret
set/get/list/delete/rotate/namespaces`). Внешнее хранилище не требуется.

**Построение с приоритетом безопасности** — обширный внутренний набор тестов плюс независимая
проверка двумя внешними аудиторами кода (DeepSeek V3 и xAI Grok). Заголовки
безопасности, защита от CSRF, ограничение скорости и санитизация входных данных на каждой поверхности API.
Предотвращение SQL-инъекций с параметризованными запросами везде.

**Целостность резервных копий** — архивы резервных копий с подписью HMAC с защитой от zip-slip,
предотвращением zip-бомб и проверкой контрольной суммы манифеста при восстановлении.

## Справочник по конфигурации

Минимальный `divisions.yaml` для начала работы:

```yaml
organization:
  name: "my-org"
  tier: 1

divisions:
  - name: operations
    tier: 2
    agents:
      - name: ops-agent
        provider: anthropic
        model: claude-haiku-4-5-20251001
        division: operations
        budget:
          per_task_usd: 0.25
          per_month_usd: 25.00

governance:
  stage0:
    enabled: true
    forbidden_actions:
      - delete_database
      - exfiltrate_data
    classification:
      default_level: INTERNAL
      max_agent_level: CONFIDENTIAL
```

`sidjua apply` подготавливает полную инфраструктуру из этого файла. Запустите повторно
после изменений — команда идемпотентна.

См. [docs/SIDJUA-APPLY-TECH-SPEC-V1.md](docs/SIDJUA-APPLY-TECH-SPEC-V1.md)
для полной спецификации всех 10 шагов подготовки.

## REST API

SIDJUA REST API работает на том же порту, что и панель управления:

```bash
sidjua server start --port 4200 --api-key $SIDJUA_API_KEY
```

Ключевые конечные точки:

```
GET  /api/v1/health          # Публичная проверка состояния (без аутентификации)
GET  /api/v1/info            # Метаданные системы (с аутентификацией)
POST /api/v1/execute/run     # Отправить задание
GET  /api/v1/execute/:id/status  # Статус задания
GET  /api/v1/execute/:id/result  # Результат задания
GET  /api/v1/events          # Поток событий SSE
GET  /api/v1/audit/report    # Отчёт о соответствии
```

Все конечные точки, кроме `/health`, требуют аутентификации Bearer. Создайте ключ:

```bash
sidjua api-key generate
```

## Docker Compose

```yaml
services:
  sidjua:
    image: ghcr.io/goetzkohlberg/sidjua:latest
    ports:
      - "4200:4200"
    volumes:
      - sidjua-data:/data
    restart: unless-stopped

volumes:
  sidjua-data:
```

Или используйте прилагаемый `docker-compose.yml`, который добавляет именованные тома для конфигурации,
журналов и рабочего пространства агента, плюс опциональный сервис Qdrant для семантического поиска:

```bash
docker compose up -d
docker compose exec sidjua sidjua init
docker compose exec sidjua sidjua apply
```

## Провайдеры

SIDJUA подключается к любому провайдеру LLM без привязки к поставщику:

| Провайдер | Модели | API-ключ |
|-----------|--------|----------|
| Anthropic | Claude Sonnet 4.6, Haiku 4.5, Opus 4.6 | `ANTHROPIC_API_KEY` |
| Google | Gemini 2.0, Gemini 1.5 Pro | `GOOGLE_API_KEY` |
| Groq | Llama 3.x, Mixtral | `GROQ_API_KEY` (бесплатный уровень) |
| Cloudflare Workers AI | Llama, Mistral, Phi | `CF_ACCOUNT_ID` + `CF_TOKEN` |
| Ollama | Любая локальная модель | Ключ не нужен (локальный) |
| Совместимый с OpenAI | Любая конечная точка | Пользовательский URL + ключ |

```bash
# Добавить ключ провайдера
sidjua key set groq gsk_...

# Список доступных провайдеров и моделей
sidjua provider list
```

## Переводы

SIDJUA доступен на 26 языках. Английский и немецкий поддерживаются основной командой. Все остальные переводы генерируются ИИ и поддерживаются сообществом.

**Документация:** Этот README и [Руководство по установке](docs/INSTALLATION.md) доступны на всех 26 языках. Смотрите выбор языка вверху этой страницы.

| Регион | Языки |
|--------|-------|
| Америка | Английский, Испанский, Португальский (Бразилия) |
| Европа | Немецкий, Французский, Итальянский, Нидерландский, Польский, Чешский, Румынский, Русский, Украинский, Шведский, Турецкий |
| Ближний Восток | Арабский |
| Азия | Хинди, Бенгальский, Филиппинский, Индонезийский, Малайский, Тайский, Вьетнамский, Японский, Корейский, Китайский (упрощённый), Китайский (традиционный) |

Нашли ошибку в переводе? Откройте GitHub Issue с:
- Языком и кодом локализации (например, `fil`)
- Неправильным текстом или ключом из файла локализации (например, `gui.nav.dashboard`)
- Правильным переводом

Хотите поддерживать язык? Смотрите [CONTRIBUTING.md](CONTRIBUTING.md#translations) — мы используем модель мейнтейнера для каждого языка.
